# Project Guard — 검증 워크플로우 원칙

## 하드 강제 (hooks가 자동 실행)
- 파일 수정(Edit/Write) 후 해당 파일의 lint가 자동 실행됩니다. 에러가 있으면 즉시 수정하세요.
- git commit 전에 staged 파일의 lint + type-check + test가 자동 실행됩니다. 실패 시 커밋이 차단됩니다.

## 소프트 강제 (반드시 따를 것)
- 새 함수/모듈 작성 시 테스트도 함께 작성하세요.
- 테스트 없이 구현 완료를 주장하지 마세요.
- 커밋 전에 관련 테스트가 통과하는지 직접 확인하세요.

## SessionStart 컨텍스트 활용
- 세션 시작 시 프로젝트 상태(스택, scripts, TODO/FIXME, 미커밋 변경)가 제공됩니다.
- FIXME/HACK이 있으면 관련 작업 시 우선 해결을 고려하세요.
- 실패한 테스트 이력이 있으면 새 기능 전에 먼저 수정을 제안하세요.
- 미커밋 변경이 많으면 커밋 정리를 먼저 권장하세요.

## 새 프로젝트 감지
- SessionStart 시 프로젝트에 검증 워크플로우 설정이 없으면 알림이 표시됩니다.
- `/project-guard` 스킬을 실행하여 프로젝트별 lint, test, type-check 명령을 설정하세요.
